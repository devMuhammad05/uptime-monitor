# Uptime Monitor API

A Laravel 13 REST API for tracking the availability of URLs. You register a URL, the system polls it on a schedule, records each check result, and sends email alerts when a site goes down or recovers.

---

## How it works

1. The Laravel scheduler runs every minute and dispatches a `CheckMonitorJob` for every monitor whose `last_checked_at` is past its `check_interval`.
2. The job makes an HTTP GET request to the monitor's URL (10-second timeout, redirects not followed).
3. The result is recorded as a `MonitorCheck` row with the HTTP status code, response time, and whether the site was up (`2xx`/`3xx` = up, anything else or a connection failure = down).
4. Consecutive failures are counted. When they reach the monitor's `threshold`, the monitor is marked **down**.
5. A successful check resets the counter and marks the monitor **up**.
6. When a monitor transitions **to down** or **back up**, an email alert is sent to the address in `MONITOR_ALERT_EMAIL`.
7. `uptime_percentage` is recalculated after every check as `(up checks / total checks) × 100`.

## Requirements

- PHP 8.4+
- Composer
- A relational database: SQLite (zero config), MySQL 8+, or PostgreSQL 15+

## Setup

### 1. Install dependencies

```bash
composer install
```

### 2. Configure the environment

```bash
cp .env.example .env
php artisan key:generate
```

Open `.env` and set the values for your environment. The defaults already work for a local SQLite setup — the minimum you need to change is `MONITOR_ALERT_EMAIL` if you want alert emails.

### 3. Database

**SQLite (default — no server needed)**

The `.env.example` already points to SQLite. Just run migrations:

```bash
php artisan migrate
```

**MySQL**

```env
DB_CONNECTION=mysql
DB_HOST=127.0.0.1
DB_PORT=3306
DB_DATABASE=uptime_monitor
DB_USERNAME=your_user
DB_PASSWORD=your_password
```

**PostgreSQL**

```env
DB_CONNECTION=pgsql
DB_HOST=127.0.0.1
DB_PORT=5432
DB_DATABASE=uptime_monitor
DB_USERNAME=your_user
DB_PASSWORD=your_password
```

Then run:

```bash
php artisan migrate
```

### 4. Alert email

Set the address that receives up/down notifications:

```env
MONITOR_ALERT_EMAIL=you@example.com
```

If this is empty or unset, notifications are silently skipped — the system still monitors, it just does not send mail.

Configure your mail driver via the `MAIL_*` variables. For local development the default `log` driver writes mail output to `storage/logs/laravel.log` — no SMTP server needed:

```env
MAIL_MAILER=log
```

For production with an SMTP server:

```env
MAIL_MAILER=smtp
MAIL_HOST=smtp.example.com
MAIL_PORT=587
MAIL_USERNAME=your_user
MAIL_PASSWORD=your_password
MAIL_ENCRYPTION=tls
MAIL_FROM_ADDRESS=noreply@example.com
MAIL_FROM_NAME="Uptime Monitor"
```

### 5. Queue worker

Jobs run on the queue. Start a worker so checks are actually processed:

```bash
php artisan queue:listen
```

The `.env.example` defaults to `QUEUE_CONNECTION=database`. The `jobs` table is created by the migration that ships with the project, so no additional setup is needed.

For production, run the worker under a process supervisor (Supervisor, systemd) so it restarts on crash:

```ini
# /etc/supervisor/conf.d/uptime-monitor-worker.conf
[program:uptime-monitor-worker]
command=php /path/to/project/artisan queue:work --sleep=3 --tries=1
autostart=true
autorestart=true
```

### 6. Scheduler

The scheduler must run continuously for monitors to be polled. During development:

```bash
php artisan schedule:work
```

This runs every minute in the foreground. For production, add a single cron entry on the server:

```cron
* * * * * cd /path/to/project && php artisan schedule:run >> /dev/null 2>&1
```

### 7. Start the HTTP server (development)

```bash
php artisan serve
```

The API will be available at `http://localhost:8000/api`.

**Development shortcut** — starts the HTTP server, queue worker, and Vite together:

```bash
composer dev
```

The scheduler still needs to run separately with `php artisan schedule:work`.

## Environment variables reference

| Variable | Required | Default | Description |
|---|---|---|---|
| `APP_KEY` | yes | — | Generated by `php artisan key:generate`. |
| `DB_CONNECTION` | no | `sqlite` | Database driver (`sqlite`, `mysql`, `pgsql`). |
| `DB_DATABASE` | no | `database/database.sqlite` | Database name or SQLite file path. |
| `MONITOR_ALERT_EMAIL` | no | — | Email address for up/down alerts. Omit to disable notifications. |
| `MAIL_MAILER` | no | `log` | Mail driver. Use `log` for development, `smtp` for production. |
| `MAIL_FROM_ADDRESS` | no | — | Sender address for alert emails. |
| `QUEUE_CONNECTION` | no | `database` | Queue driver. `database` works out of the box with no extra services. |

## Running tests

The test suite uses Pest and runs against an in-memory SQLite database — no server, queue, or mail setup needed:

```bash
composer test
```

This runs a code-style check (Laravel Pint) followed by the full test suite. To skip the style check and run tests only:

```bash
php artisan test
```

**What is covered:**

- `POST /api/monitors` — creation, default values, all validation rules (required URL, HTTP/HTTPS only, uniqueness, interval and threshold bounds).
- `GET /api/monitors` — empty list, field structure, correct status values, pagination.
- `GET /api/monitors/{id}/history` — 404 for unknown monitors, field structure, descending order, pagination (`page`, `per_page`, cap at 100, default of 15), timeout representation (`status_code=0`, `response_time_ms=null`), isolation between monitors.
- `CheckMonitorJob` — HTTP check recording (200, 301, 500, connection failure), threshold and consecutive-failure logic, uptime percentage, `last_checked_at` updates, all notification transitions (pending→down, up→down, down→up, stays up, stays down, below threshold, no alert email configured), deleted monitor guard.
- Rate limiting — POST capped at 10/min, GET endpoints capped at 60/min.

## Middleware

### ForceJsonResponse

Sets `Accept: application/json` on every incoming request so Laravel always returns JSON — for validation errors, 404s, and unhandled exceptions — regardless of what the client sends. Registered globally on the `api` group in `bootstrap/app.php`.

### Rate limiting

Read endpoints (`GET /api/monitors`, `GET /api/monitors/{id}/history`) are limited to **60 requests per minute**. The create endpoint (`POST /api/monitors`) is limited to **10 requests per minute**. Clients that exceed the limit receive a `429 Too Many Requests` response.

## API Reference

All endpoints are prefixed with `/api`. Every response is JSON with this envelope:

### Create a monitor

```
POST /api/monitors
Content-Type: application/json
```

**Request body**

| Field | Type | Required | Notes |
|---|---|---|---|
| `url` | string | yes | Must be a valid HTTP or HTTPS URL. Must be unique. |
| `check_interval` | integer | no | Minutes between checks. Between 1 and 60. Defaults to `5`. |
| `threshold` | integer | no | Consecutive failures before marking the monitor as down. Minimum 1. Defaults to `3`. |

**Example**

```json
{
  "url": "https://example.com",
  "check_interval": 10,
  "threshold": 2
}
```

**Response 201**

```json
{
  "status": "success",
  "message": "Monitor created successfully",
  "data": {
    "id": 1,
    "url": "https://example.com",
    "check_interval": 10,
    "threshold": 2,
    "status": "pending",
    "consecutive_failures": 0,
    "uptime_percentage": null,
    "last_checked_at": null,
    "created_at": "2026-05-17T10:00:00.000000Z"
  }
}
```

**Response 422** — validation failure

```json
{
  "message": "The url field is required.",
  "errors": {
    "url": ["The url field is required."]
  }
}
```

### Get monitor history

```
GET /api/monitors/{id}/history
```

Returns paginated check results for a given monitor, ordered from most recent to oldest.

**Query parameters**

| Parameter | Default | Notes |
|---|---|---|
| `page` | 1 | Page number. |
| `per_page` | 15 | Results per page. Maximum 100. |

**Response 200**

```json
{
  "status": "success",
  "message": "Monitor history retrieved successfully",
  "data": [
    {
      "id": 1,
      "monitor_id": 1,
      "status_code": 200,
      "response_time_ms": 142,
      "is_up": true,
      "checked_at": "2026-05-17T10:05:00.000000Z"
    },
    {
      "id": 2,
      "monitor_id": 1,
      "status_code": 0,
      "response_time_ms": null,
      "is_up": false,
      "checked_at": "2026-05-17T10:00:00.000000Z"
    }
  ],
  "meta": {
    "current_page": 1,
    "per_page": 15,
    "total": 2,
    "last_page": 1
  }
}
```

A `status_code` of `0` and a `null` `response_time_ms` indicate a connection failure or timeout.

**Response 404**

```json
{
  "message": "Monitor not found."
}
```

## Monitor status values

| Status | Meaning |
|---|---|
| `pending` | Created but not yet checked. |
| `up` | The URL is responding successfully. |
| `down` | The URL has failed consecutively at least as many times as `threshold`. |
